프로그램 개발 통제

가이드라인은 새로운 시스템의 개발과 도입이 적절한 승인, 개발 방법론, 테스트, 문서화, 운영 전환 통제를 통해 관리되어야 한다고 설명합니다. 프로그램 개발 통제는 새 시스템이 처음부터 잘못 설계되어 회계정보를 왜곡하는 상황을 막기 위한 영역입니다.

핵심 점검 항목

• 신규 시스템 도입 승인

• 통제가 내재된 개발 방법론 적용

• 기존 통제의 재설계 여부 검토

• 현업과 전산부서의 테스트

• 문서화와 운영환경 이전 통제

1. 왜 프로그램 개발 통제가 필요한가

새 시스템은 편의만 높이는 도구가 아니라 거래 입력, 처리, 계산, 보고를 바꾸는 구조적 변화입니다. 만약 설계 단계에서 승인 절차가 빠지거나, 기존 통제가 누락되거나, 테스트 없이 도입되면 오류는 운영 초기에 대량으로 반복될 수 있습니다. 따라서 개발단계에서부터 회계통제를 함께 설계하는 것이 중요합니다.

새 시스템이 들어오면 기존 통제가 그대로 유지되는지, 아니면 새로 설계되어야 하는지를 반드시 따져야 합니다.

2. 가이드라인이 제시하는 개발 통제

통제 항목 설명
경영진 승인 새로운 시스템 개발과 도입은 적절한 승인 후 진행
개발 방법론 통제가 내재된 절차에 따라 개발 수행
기존 통제 반영 영향받는 기존 통제를 수정 또는 재설계
테스트 수행 현업과 전산부서가 함께 적절한 테스트 수행
문서화 시스템, 사용자, 관련 통제에 대한 문서 확보
운영 전환 통제 운영환경 이전 시 접근통제와 승인 적용
데이터 이전 완전성 이전 데이터의 누락·중복·손상 여부 확인

3. 감사 관점의 질문

  • 신규 시스템 도입은 누가 승인했는가
  • 회계 관련 요구사항이 개발서에 반영되었는가
  • 사용자 테스트 기록이 남아 있는가
  • 운영 전환 전에 권한과 보안 설정이 검토되었는가
  • 데이터 이전 결과를 누가 확인했는가

4. 자주 발생하는 미비점

실무에서는 개발 일정에 쫓겨 테스트를 축소하거나, 개발자 중심으로만 검토하고 현업 검증을 약하게 하는 경우가 있습니다. 또 기존 수작업 통제가 시스템에서 사라졌는데 대체통제를 만들지 못하는 경우도 많습니다. 이러한 미비점은 운영 초기에 오류가 집중되는 원인이 됩니다.

  • 기존 통제가 사라지는 경우 대체통제가 설계되는가
