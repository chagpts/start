전산감사 절차와 방법론

전산감사는 기업의 정보시스템, 데이터, 접근권한, 업무 프로세스, 보안 통제 등이 정확하고 안전하게 운영되고 있는지를 점검하는 감사 활동입니다. 회계 시스템, ERP, 그룹웨어, 데이터베이스, 클라우드 환경 등 업무가 전산화된 조직에서는 전산감사가 내부통제와 재무보고 신뢰성 확보에 매우 중요합니다.

1. 전산감사란 무엇인가

전산감사는 IT 환경에서 발생할 수 있는 위험을 식별하고, 그 위험을 줄이기 위한 통제가 제대로 설계되고 운영되는지 확인하는 절차입니다. 단순히 컴퓨터 시스템을 점검하는 것이 아니라, 업무 프로세스와 데이터 흐름이 신뢰할 수 있는지 확인하는 활동입니다.

전산감사의 핵심은 “시스템이 안전한가”만 보는 것이 아니라, “시스템을 통해 생성되는 업무 결과와 회계 데이터가 신뢰할 수 있는가”를 확인하는 것입니다.

2. 전산감사의 목적

목적 설명
데이터 신뢰성 확보 시스템에서 생성되는 데이터가 정확하고 완전한지 확인
접근권한 통제 확인 허가된 사용자만 시스템과 데이터에 접근하는지 점검
변경관리 검토 프로그램 변경, 시스템 설정 변경이 승인 절차를 거쳤는지 확인
업무 연속성 확보 장애, 백업, 복구 체계가 적절히 운영되는지 확인
부정 및 오류 예방 비정상 거래, 권한 남용, 데이터 조작 가능성을 점검

3. 전산감사 전체 절차

전산감사는 일반적으로 계획, 위험 평가, 통제 이해, 통제 테스트, 데이터 분석, 발견사항 정리, 보고 및 사후관리의 순서로 진행됩니다.

감사 계획 → 시스템 이해 → 위험 평가 → 통제 설계 평가 → 운영 효과성 테스트 → 데이터 분석 → 발견사항 보고 → 개선 조치 확인

4. 감사 계획 수립

감사 계획 단계에서는 감사 대상 시스템, 범위, 기간, 목적, 필요한 자료를 정합니다. 예를 들어 ERP의 매출 모듈을 볼 것인지, 구매 모듈을 볼 것인지, 또는 사용자 권한과 변경관리만 볼 것인지 범위를 명확히 해야 합니다.

  • 감사 대상 시스템 선정
  • 감사 목적과 범위 정의
  • 주요 이해관계자 확인
  • 필요 자료 목록 작성
  • 감사 일정과 인터뷰 계획 수립

5. 시스템 이해 및 업무 프로세스 파악

전산감사는 시스템만 보는 것이 아니라 업무 흐름을 함께 이해해야 합니다. 주문, 승인, 입력, 처리, 검토, 보고가 어떤 시스템에서 이루어지는지 파악해야 어디에 위험이 있는지 판단할 수 있습니다.

확인 항목 예시 질문
업무 흐름 거래는 어디서 시작되고 어떤 승인 절차를 거치는가?
시스템 연계 ERP, 은행, 쇼핑몰, 회계 프로그램 간 데이터가 어떻게 이동하는가?
입력 권한 누가 거래를 생성, 수정, 삭제할 수 있는가?
승인 절차 시스템상 승인권자와 실제 업무 책임자가 일치하는가?
보고서 생성 재무 보고에 사용되는 데이터는 어떤 시스템에서 추출되는가?

6. 위험 평가

위험 평가는 어떤 오류나 부정이 발생할 수 있는지 식별하는 단계입니다. 전산감사에서는 권한 남용, 데이터 변경, 승인 누락, 인터페이스 오류, 백업 실패, 로그 미보관 등이 주요 위험입니다.

위험 유형 설명 예시
접근권한 위험 권한이 과도하거나 퇴사자 계정이 남아 있는 경우 퇴사자가 ERP에 로그인 가능
변경관리 위험 승인 없이 프로그램이나 설정이 변경되는 경우 매출 계산 로직 무단 변경
데이터 무결성 위험 데이터가 누락, 중복, 변조되는 경우 인터페이스 오류로 거래 일부 누락
업무 분장 위험 한 사람이 생성, 승인, 지급까지 모두 수행하는 경우 구매 담당자가 지급 승인까지 가능
운영 위험 백업, 장애 대응, 로그 관리가 미흡한 경우 장애 발생 시 복구 불가

7. IT 일반통제

IT 일반통제는 여러 시스템에 공통적으로 적용되는 기본 통제입니다. 시스템 접근권한, 프로그램 변경관리, 운영관리, 백업 및 복구 등이 여기에 포함됩니다. 일반통제가 약하면 개별 업무 시스템의 결과도 신뢰하기 어려워집니다.

통제 영역 점검 내용
접근권한 관리 신규 계정 승인, 권한 변경 승인, 퇴사자 권한 회수 여부
비밀번호 정책 복잡도, 변경 주기, 잠금 정책, 공유 계정 사용 여부
변경관리 프로그램 수정 요청, 승인, 테스트, 배포 기록 존재 여부
운영관리 배치 작업, 장애 처리, 로그 모니터링, 운영 절차 문서화 여부
백업 및 복구 백업 주기, 보관 위치, 복구 테스트 수행 여부

8. 응용통제

응용통제는 특정 업무 시스템이나 거래 처리 과정에 적용되는 통제입니다. 예를 들어 매출 주문 승인, 구매 승인 한도, 중복 전표 방지, 필수 입력값 검증, 자동 계산 로직 등이 응용통제에 해당합니다.

통제 유형 예시
입력 통제 필수값 누락 방지, 금액 형식 검증, 거래처 코드 검증
처리 통제 세금 자동 계산, 승인 한도 적용, 중복 거래 차단
출력 통제 보고서 접근 제한, 출력 이력 관리, 보고서 기준 검토
인터페이스 통제 시스템 간 데이터 전송 건수와 금액 대사

9. 감사 증거 수집 방법

전산감사에서는 인터뷰, 문서 검토, 시스템 설정 확인, 로그 분석, 샘플 테스트, 재수행, 데이터 분석 등을 통해 감사 증거를 수집합니다. 증거는 충분하고 적합해야 하며, 발견사항을 뒷받침할 수 있어야 합니다.

  • 담당자 인터뷰
  • 정책 및 절차서 검토
  • 시스템 권한 목록 확인
  • 변경관리 티켓 및 승인 기록 검토
  • 로그인 이력, 변경 이력, 작업 로그 분석
  • 데이터 추출 후 이상 거래 분석

10. 데이터 분석 기반 감사

전산감사에서는 전체 데이터를 분석하여 이상 징후를 찾을 수 있습니다. 샘플만 보는 방식보다 더 넓은 범위를 확인할 수 있으며, 중복 거래, 비정상 금액, 야간 입력, 승인 누락 등을 탐지하는 데 효과적입니다.

분석 주제 분석 예시
중복 거래 분석 동일 거래처, 동일 금액, 동일 날짜의 중복 전표 확인
권한 이상 분석 관리자 권한 보유자, 퇴사자 계정, 장기 미사용 계정 확인
시간대 분석 야간, 주말, 휴일에 입력된 거래 확인
금액 이상 분석 승인 한도 바로 아래 금액의 반복 거래 확인
변경 이력 분석 마스터 데이터 수정, 계좌번호 변경, 거래처 정보 변경 확인 
예시 분석 로직

1. 동일 거래처 + 동일 금액 + 동일 날짜 거래 추출
2. 승인자와 입력자가 같은 거래 추출
3. 퇴사일 이후 로그인 기록이 있는 계정 확인
4. 주말 또는 심야 시간에 생성된 전표 확인
5. 변경 승인 없이 수정된 프로그램 배포 기록 확인

11. 표본 테스트와 전수 분석

전산감사에서는 표본 테스트와 전수 분석을 함께 사용할 수 있습니다. 표본 테스트는 특정 거래나 통제를 선택하여 깊게 검토하는 방식이고, 전수 분석은 전체 데이터를 대상으로 이상 패턴을 찾는 방식입니다.

방식 특징 적합한 상황
표본 테스트 일부 거래를 선택해 승인, 증빙, 처리 과정을 자세히 확인 통제 운영 증거를 깊게 확인할 때
전수 분석 전체 데이터를 대상으로 중복, 누락, 이상 거래를 탐지 데이터 기반 이상 징후를 찾을 때
재수행 감사인이 직접 계산이나 통제 절차를 다시 수행 자동 계산 로직이나 승인 규칙을 검증할 때

12. 발견사항 정리

14. 사후관리

전산감사는 보고서 작성으로 끝나지 않습니다. 발견사항에 대한 개선 조치가 실제로 완료되었는지 확인해야 합니다. 권한 회수, 정책 변경, 시스템 설정 수정, 로그 모니터링 도입 등 후속 조치가 실행되어야 감사 효과가 완성됩니다.

좋은 전산감사는 문제를 찾는 데서 끝나지 않고, 조직의 시스템과 데이터 신뢰성을 실제로 높이는 데까지 이어져야 합니다.
