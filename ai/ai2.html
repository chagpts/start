AI OCR Accounting Platform (Local-Only, No Paid APIs)

This plan outlines a 24-week build for an end-to-end accounting automation system: Receipt/PDF upload → OCR → AI-assisted extraction & classification → draft journal entries → approval/posting → ledger updates → automated financial statements (BS/IS/CF).

Target Environment: Ubuntu + Docker + Node.js + MongoDB + Ollama (local LLM) + Tesseract OCR
Cost Policy: No paid APIs. No external AI inference. Everything runs locally.

English Reference Notes — “What if We Have Zero Training Data?”

Core point: You can start with training data = 0 and still deliver a usable MVP by combining: (1) OCR + rule-based extraction with confidence scoring, (2) accounting review only for low-confidence fields, (3) merchant-to-account mappings as the primary classifier, (4) local LLM only for ambiguous cases, and (5) “draft-only + approval required” posting workflow, while storing all corrections/rejections to improve rules and mappings over time.

1) Why This Works Without Training

  • Extraction is mostly deterministic: totals, dates, VAT, and payment indicators can be found via regex + validation rules.
  • Classification can be policy-driven: merchant mapping tables often outperform LLMs for stability and consistency.
  • LLM is used as a helper, not a source of truth: only for “unknown” or ambiguous cases, and always behind schema validation.
  • Safety comes from workflow: the system generates draft journal entries; humans approve and post.
  • Accuracy improves through operations: every correction becomes a “signal” to refine rules, mappings, and prompts.

2) MVP Flow (Safe When Training Data = 0)

Step What the System Does Why It’s Safe
OCR Convert PDF→image (Poppler), run Tesseract, store raw text. No AI decisions; just text extraction.
Rule-based Field Extraction Regex + validation → merchant/date/total/VAT/payment + confidence per field. Deterministic; confidence indicates uncertainty.
Accounting Review (Selective) Only route low-confidence fields to review UI; store corrections. Humans fix uncertainty; reduced workload.
Classification (Tiered) Tier 1: merchant mapping table. Tier 2: local LLM only when needed. Stable rules first; LLM is fallback.
Draft Journal Entry Create a balanced JE draft using templates; never auto-post. Prevents incorrect posting; audit-ready.
Approve & Post Accountant approves; system posts if period open; logs everything. Controlled posting + period locks.
Improve Loop Store corrections/rejections; update mappings/rules/prompts weekly. Continuous accuracy improvement without model training.

Example OCR text (Starbucks-like receipt):

AMERICANO T  4,500
TOTAL        4,500
VAT          409
CARD         5678

  "merchantName":  { "value": "STARBUCKS KOREA", "confidence": 0.92, "source": "top_lines" },
  "transactionAt": { "value": "2026-02-20T14:33:00+09:00", "confidence": 0.85, "source": "regex_ymd_time" },
  "totalAmount":   { "value": 4500, "confidence": 0.95, "source": "keyword_total" },
  "vatAmount":     { "value": 409,  "confidence": 0.90, "source": "keyword_vat" },
  "paymentMethod": { "value": "CARD","confidence": 0.80, "source": "keyword_card" },
  "currency":      { "value": "KRW", "confidence": 0.99, "source": "default" }
}

3.2 Review Routing Rule (Only When Needed)

// Example routing logic
needsReview =
  (total.confidence   < 0.75) ||
  (transaction.confidence < 0.75) ||
  (merchant.confidence < 0.70) ||
  (vat.value == null) ||
  (payment.confidence < 0.70);

3.3 Merchant Mapping Table (Tier 1 Classifier)

{
  "merchantPattern": "STARBUCKS|스타벅스",
  "defaultDebitAccount": "5100_MEALS",
  "defaultCreditAccountByPayment": {
    "CARD": "2105_CARD_CLEARING",
    "CASH": "1101_CASH",
    "TRANSFER": "1102_BANK"
  },
  "vatRule": "DEDUCTIBLE_IF_VAT_PRESENT",
  "confidenceBoost": 0.25
}

3.4 LLM Fallback (Tier 2 Only When Ambiguous)

  • Call the local LLM only if merchant mapping fails or required fields are uncertain.
  • Force strict JSON output, validate against schema, retry on invalid output; otherwise route to manual review.

4) What Must Be Stored to Improve Without Training

  • Extraction corrections: field-level “from → to”, user, timestamp, reason.
  • Classification overrides: which account/category was changed and why.
  • Rejection reasons: codes such as WRONG_MERCHANT / WRONG_VAT / WRONG_ACCOUNT / MISSING_DATE.

Example: classification override record

{
  "documentId": "doc_002",
  "override": {
    "from": { "debitAccount": "5100_MEALS" },
    "to":   { "debitAccount": "5203_OFFICE_SUPPLIES" },
    "reasonCode": "WRONG_ACCOUNT",
    "by": "u_hana",
    "at": "2026-02-21T04:02:00Z"
  }
}

5) Practical Success Criteria (Realistic Expectations)

  • MVP success: reduces manual typing by extracting fields and generating balanced drafts.
  • Safety: no auto-posting; approvals + audit trail + period locks are mandatory.
  • Accuracy growth: merchant mapping coverage improves weekly; LLM is gradually used less.

Role Play — Dev, Accounting, PM Alignment (Based on the Notes Above)

Context: The team is concerned about starting with zero training data. They agree on a safe MVP: rule-based extraction + selective review + merchant mapping first + LLM only for ambiguous cases + draft-only entries + approval required.
PM (Mina):

We keep hearing the same concern: “We have no training data.” Before we commit, I want a clear plan for how the MVP works safely from day one, and how it improves over time.

Developer (Alex):

The MVP doesn’t depend on training. We’ll start deterministic: OCR with Tesseract, then rule-based extraction for merchant/date/total/VAT/payment. Every extracted field carries a confidence score. Only low-confidence fields go to accounting review.

Accounting Lead (Jisoo):

I’m okay with that if the review is selective and fast. But I don’t want a system that posts entries automatically. Drafting is fine; posting must be a deliberate action with audit logs.

Developer (Alex):

Agreed. We’ll do “draft-only” journal entries by default. The system generates a balanced draft using templates, but it never posts without an accountant’s approval. We’ll also enforce period locks so nothing posts to a closed month.

IT/Security (Daniel):

The other risk is unreliable AI. If you use an LLM at all, we need predictable outputs, schema validation, and no external calls. Everything must run locally.

Developer (Alex):

The LLM is strictly a fallback. First classification tier is a merchant mapping table that accounting can maintain: “STARBUCKS → Meals”, “Korean Air → Travel”, etc. Only when mapping fails or the document is ambiguous do we call the local LLM. Even then, it must output strict JSON, validated by schema, with retries; otherwise it routes back to manual review.

Senior Accountant (Hana):

That merchant mapping table is the key. If it’s easy to update, our accuracy will jump quickly. But I also want metrics: how often we correct totals, how often we override the suggested account, and what the auto-approval rate is over time.

Developer (Alex):

We’ll log everything: extraction confidence, corrections (from → to), classification confidence, overrides, rejection reasons, and end-to-end processing time. Each correction becomes a signal: if a merchant is frequently overridden, we update the mapping; if VAT is frequently missing, we improve the extraction rules.

PM (Mina):

Let’s test with one receipt. Starbucks-like: total 4,500 KRW, VAT 409, paid by corporate card. What happens in the system?

Developer (Alex):

OCR extracts text. Rule-based extraction finds merchant/date/total/VAT/payment with confidence scores. Because VAT is present and total is high-confidence, it likely skips review. Merchant mapping matches STARBUCKS, so classification is stable. We generate a draft JE: debit Meals net, debit Input VAT (if deductible), credit Card Clearing total. Then accounting reviews and approves; only then can it post.

Accounting Lead (Jisoo):

Great. One more constraint: VAT deductibility depends on our policy and document type. If VAT is missing or unclear, the system must not guess silently. It must flag “VAT unknown” and require review.

Developer (Alex):

We’ll implement that explicitly. If VAT is missing, or policy conditions aren’t met, we set VATUnknown=true and route to review. We’ll keep the default conservative: require manual confirmation unless a known merchant mapping indicates consistent VAT behavior.

IT/Security (Daniel):

Reliability wise, OCR and LLM can fail. We need retries, idempotency, and no duplicates—especially around posting.

Developer (Alex):

We’ll process documents via a queue (BullMQ/Redis recommended) with idempotent steps. Posting requires a unique posting key so it cannot run twice even if a worker restarts. Failures are logged and routed to a retry or manual review state.

PM (Mina):

So the MVP definition is clear: no training required, no auto-posting, and accuracy improves via mapping + corrections. Let’s lock that scope and build Month 2 (OCR + extraction + review) first, then Month 3 (mapping + LLM fallback + draft JE).
