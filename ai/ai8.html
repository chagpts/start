회계 시스템 연계와 자동분개 설계

AI OCR이 문서를 읽는 것만으로는 회계 프로그램이 완성되지 않습니다. 실제 업무 가치는 OCR 결과가 회계 시스템으로 연결되어 전표 생성, 거래처 매핑, 계정과목 추천, 검토 승인, 증빙 보관까지 이어질 때 발생합니다. 즉, OCR은 시작점이고 핵심은 회계 처리 흐름과 연결되는 구조를 만드는 것입니다.

이번 과정의 핵심 내용

• OCR 결과를 회계 데이터로 변환하는 방법

• 자동분개 규칙 설계 원칙

• 거래처, 계정과목, 세금코드 매핑 구조

• 검토 승인 흐름과 예외 처리 구조

1. OCR 결과와 회계 데이터는 다르다

OCR 결과는 문서에서 읽은 값입니다. 하지만 회계 시스템은 단순 텍스트가 아니라 전표일자, 거래처 코드, 계정과목, 차변·대변, 세금코드, 부서, 프로젝트 코드 같은 구조화된 회계 데이터를 필요로 합니다. 예를 들어 영수증에서 “가맹점명”을 읽었다고 해서 바로 거래처 코드가 생기는 것은 아닙니다. 중간에 거래처 마스터 매핑과 규칙 검증이 필요합니다.

문자를 읽는 단계와 회계 처리를 만드는 단계는 다르며, 두 단계 사이의 변환 설계가 시스템 완성도를 결정합니다.

2. 자동분개는 어떻게 설계하는가

자동분개는 문서 종류, 거래처 특성, 금액 구조, 세금 정보, 부서 정보 등을 바탕으로 분개 후보를 생성하는 기능입니다. 처음부터 모든 분개를 완전 자동화하려 하기보다 반복성이 높은 거래부터 규칙 기반으로 자동화하고, 불확실한 경우는 추천 형태로 제시하는 것이 현실적입니다. 예를 들어 정기적으로 들어오는 통신비 청구서는 거래처와 항목이 일정하므로 특정 비용 계정으로 자동 추천하기 쉽습니다. 반면 예외가 많은 지출은 사람 검토가 필요합니다.

자동분개 요소 설명
문서 유형 세금계산서, 영수증, 거래명세서 등에 따라 분개 흐름이 달라짐
거래처 매핑 문서의 거래처명을 ERP 거래처 코드와 연결
계정과목 추천 거래 유형과 과거 전표 패턴을 바탕으로 추천
세금코드 결정 부가세 포함 여부, 적격증빙 여부 반영
검토 상태 자동확정, 검토필요, 보류 등 상태 관리

3. 거래처와 마스터 데이터 연계

회계 프로그램은 OCR만으로 끝나지 않고 기존 마스터 데이터와 연결되어야 합니다. 거래처명이 문서마다 조금씩 다르게 찍힐 수 있으므로 문자열 일치만으로는 한계가 있습니다. 사업자등록번호, 계좌번호, 과거 거래 이력, 별칭 사전 등을 활용해 매핑 정확도를 높이는 구조가 필요합니다. 또한 신규 거래처가 발견되면 자동 등록이 아니라 검토 후 등록하도록 통제하는 것이 안전합니다.

4. 예외 처리는 반드시 분리해야 한다

모든 문서가 자동분개에 성공하는 것은 아닙니다. 거래처를 찾지 못하거나, 금액 검증이 맞지 않거나, 중복 전표 가능성이 있거나, 문서 품질이 낮은 경우에는 자동으로 확정하면 안 됩니다. 이런 건들은 예외 큐로 분리하고, 담당자가 확인 후 수정할 수 있는 화면과 절차를 둬야 합니다. 예외 처리를 별도로 설계하지 않으면 운영 단계에서 오히려 수작업 부담이 커집니다.

예외 상황 필요 조치
거래처 미매핑 후보 거래처 추천 후 담당자 선택
금액 불일치 공급가액, 세액, 합계금액 재검토
문서 중복 가능성 기존 전표와 비교 후 등록 제한
문서 품질 저하 원본 재업로드 또는 수기 입력 전환
계정과목 불확실 추천만 제공하고 확정은 담당자 승인

5. 자동화 수준은 단계적으로 높여야 한다

  • 1단계: OCR 결과 추출과 검수 화면 제공
  • 2단계: 거래처, 세금코드, 계정과목 추천 제공
  • 3단계: 규칙 기반 자동분개 생성
  • 4단계: 신뢰도 높은 문서에 한해 자동전표 등록
  • 5단계: 예외와 오류 패턴을 학습하여 자동화 범위 확대

회계 프로그램에서는 완전 자동화보다 통제 가능한 반자동화가 더 현실적이고 안전한 경우가 많습니다.

6. 역할극으로 이해하기

개발 담당자: OCR에서 거래처명과 금액을 읽었으니 바로 전표를 생성해도 될까요?

회계 담당자: 바로는 어렵습니다. 거래처 코드와 계정과목, 세금코드 검증이 더 필요합니다.

개발 담당자: 그럼 먼저 추천 형태로 보여주고 담당자가 승인하는 흐름이 좋겠네요.

운영 담당자: 반복 거래는 나중에 자동확정으로 올리고 예외 건만 검토하면 효율이 좋아질 것 같습니다.
