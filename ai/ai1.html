AI OCR Accounting Platform — 3 Month MVP Plan

This plan outlines a realistic 3-month build for a U.S.-focused AI OCR accounting web platform. The product follows a QuickBooks-style accounting structure while adding OCR, Llama-based document understanding, accounting recommendations, approval workflow, and posting integration.

Core Product Idea: Upload invoices, receipts, and utility bills → OCR extracts the document → Llama normalizes and classifies the data → accounting rules generate posting candidates → users review and approve → data is posted into a QuickBooks-compatible workflow.

Recommended MVP Stack

FastAPI PostgreSQL S3 / Object Storage Redis / Queue OCR Engine Llama Rules Engine Next.js Admin UI QuickBooks API

Product Principle: OCR and LLM assist understanding, but accounting logic is finalized by rules and approval.

Milestones at a Glance

Month Main Goal Milestone Output
Month 1 Document intake + OCR pipeline Upload → OCR → extracted JSON → document review screen
Month 2 Llama normalization + accounting recommendation Vendor detection, account suggestion, confidence scoring, editable review flow
Month 3 Approval workflow + QuickBooks posting Review → approve → post Bill/Expense/Vendor to QuickBooks-compatible flow

Project Scope — What This MVP Is

MVP Definition: A working accounting operations tool for U.S. business documents. It should support receipt and bill ingestion, OCR extraction, LLM-assisted normalization, account recommendation, approval workflow, duplicate detection, audit logging, and QuickBooks posting for core objects such as Bill, Expense, and Vendor.

1) Product Flow

  • Upload: User uploads PDF or image documents.
  • OCR: The system extracts text and key fields from invoices, receipts, and utility bills.
  • Llama Processing: The model normalizes vendor names, classifies document type, and suggests accounts.
  • Accounting Recommendation: A rules engine converts extracted data into draft accounting actions.
  • Review & Approval: Users verify, correct, and approve entries.
  • Posting: Approved entries are pushed into a QuickBooks-compatible flow.

2) Key Design Principle

Layer Responsibility Reason
OCR Read document text and structure Best used for extraction, not accounting decisions
Llama Normalize, classify, explain, recommend Good at ambiguity handling and document interpretation
Rules Engine Generate deterministic accounting actions Required for stable accounting behavior
User Approval Validate final decision before posting Ensures trust, control, and auditability

Month 1 — Foundation + OCR Pipeline

The first month focuses on building the intake and document processing pipeline. The goal is simple: users can upload files, the system runs OCR, stores the outputs, and displays structured document data in a reviewable UI.

Month 1 Deliverables

  • Document upload system for PDF and image files
  • Document storage and metadata tracking
  • OCR integration and raw extraction pipeline
  • OCR results saved as structured JSON
  • Basic document list page and detail page

Month 1 success condition

A user can upload a document and see OCR-extracted fields in the web UI. At the end of Month 1, the system does not need to make accounting decisions yet. It only needs to read, store, and display document data reliably.

Month 2 — Llama + Accounting Recommendation

The second month adds business intelligence on top of OCR. Llama is introduced to normalize vendors, identify document types, recommend account categories, and generate confidence-based accounting suggestions. Users should be able to see and edit these recommendations before anything is finalized.

Month 2 Deliverables

  • Llama integration for normalization and classification
  • Strict JSON schema output from the model
  • Vendor normalization and alias handling
  • Chart of accounts mapping and recommendation engine
  • Confidence scores and review flags
  • Editable review UI for accounting staff

Rules engine example

// Pseudo logic
if (document_type === "vendor_bill") {
  entity = "Bill";
}

if (vendor_history.same_account_ratio > 0.8) {
  suggested_account = vendor_history.most_common_account;
}

if (amount_total <= 0 || currency == null) {
  status = "needs_review";
}

if (duplicate_check == true) {
  status = "duplicate_alert";
}

Month 2 success condition

At the end of Month 2, the platform should not just read documents. It should also understand them well enough to recommend vendors, document types, and accounting categories, while allowing a human reviewer to adjust the result.

Month 3 — Approval Workflow + QuickBooks Posting

The third month turns the system into a usable accounting operations product. This is where review states, approval workflow, duplicate protection, audit logs, and posting integration are completed.

Month 3 Deliverables

  • Approval status flow: Draft → Review → Approved → Posted
  • QuickBooks OAuth and posting integration
  • Posting support for Bill, Expense, and Vendor
  • Duplicate detection rules
  • Audit log for OCR result, AI result, edits, and posting action
  • Basic metrics dashboard for confidence and review rate

Example End-to-End Flow

Example input: a U.S. office supply invoice uploaded as a PDF. 

1. User uploads invoice PDF
2. OCR extracts vendor, invoice number, date, total, line items
3. Llama normalizes "AMZN Mktp US" to "Amazon"
4. Rules engine recommends:
   - Document Type: Vendor Bill
   - Account: Office Supplies
   - QBO Entity: Bill
5. Reviewer checks and approves
6. System posts to QuickBooks-compatible workflow
7. Audit trail stores every step

Team Alignment — How the Product Should Be Built

Context: The team is aligning on a 3-month MVP for a U.S. AI OCR accounting platform. The product is based on a QuickBooks-style accounting structure, but adds OCR, Llama, rules, and approval workflow.
PM (Mina):

We only have three months, so we need a focused MVP. I want a product that looks like a real accounting operations tool, not just an OCR demo. What is the smallest version we can ship that still feels useful?

Developer (Alex):

The minimum useful version is document upload, OCR extraction, Llama-based normalization, accounting recommendation, review UI, and QuickBooks posting for core objects only. We should keep the posting scope narrow: Bill, Expense, and Vendor.

Accounting Lead (Jisoo):

That works as long as the AI does not directly post to the books. I want accounting staff to review the recommendation first. The model can suggest, but the workflow must require approval.

Developer (Alex):

Agreed. OCR reads the document, Llama interprets it, and the rules engine generates a deterministic recommendation. The final accounting action stays in draft or review state until a human approves it.

IT/Security (Daniel):

Then we also need auditability. I want a trace of the raw OCR result, the Llama output, the reviewer edits, and the final posting action. Otherwise the system will be hard to trust.

Developer (Alex):

We can make that part of the core design. Every document will keep its original file, OCR result, normalized AI result, approval history, duplicate check result, and posting log.

Senior Accountant (Hana):

I also want the recommendation layer to get smarter as we use it. For example, if Amazon bills are usually mapped to Office Supplies, the next similar document should come in with a stronger suggestion.

Developer (Alex):

Yes. We can implement vendor alias mapping and historical account suggestions in Month 2. That way the model is not working alone; it is supported by prior accounting behavior.

PM (Mina):

Then the project should be staged clearly. Month 1 reads the document, Month 2 understands the document, and Month 3 posts the approved result.

Accounting Lead (Jisoo):

That sequencing makes sense. By the end, I want the team to be able to upload a bill, review the suggested account, approve it, and push it into the accounting flow without manual re-entry.

Suggested Discussion / Next Steps

  • Should the MVP start with receipts + vendor bills only before customer invoices?
  • Who owns the initial chart of accounts mapping rules for Month 2?
  • Should duplicate detection be strict in Month 3 or advisory only?
  • Do we want the first release to support only USD + one business entity?
