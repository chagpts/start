ITGC 일반통제의 이해

ITGC는 Information Technology General Controls의 약자로, 정보시스템이 안전하고 일관되게 운영되도록 관리하는 기본 통제를 의미합니다. 회계 시스템, ERP, 데이터베이스, 클라우드, 그룹웨어처럼 중요한 업무 시스템이 신뢰할 수 있는 데이터를 만들기 위해서는 ITGC가 제대로 작동해야 합니다.

1. ITGC란 무엇인가

ITGC는 특정 업무 하나에만 적용되는 통제가 아니라, 여러 정보시스템에 공통적으로 적용되는 기본 관리 체계입니다. 예를 들어 누가 시스템에 접근할 수 있는지, 프로그램 변경은 승인받고 배포되는지, 장애가 발생했을 때 복구할 수 있는지 등을 확인합니다.

ITGC의 핵심은 “시스템이 믿을 수 있게 운영되고 있는가”를 확인하는 것입니다. ITGC가 약하면 시스템에서 생성되는 회계 데이터와 업무 데이터도 신뢰하기 어려워집니다.

2. ITGC가 중요한 이유

기업의 회계와 업무 처리는 대부분 시스템을 통해 이루어집니다. 전표 입력, 매출 인식, 구매 승인, 급여 계산, 재고 관리, 보고서 생성 등은 ERP나 회계 프로그램에 의존합니다. 따라서 시스템에 대한 접근, 변경, 운영이 제대로 통제되지 않으면 데이터 오류나 부정 위험이 커질 수 있습니다.

위험 예시
무단 접근 권한 없는 사용자가 전표를 수정하거나 삭제
권한 과다 한 사용자가 입력, 승인, 지급까지 모두 수행
무단 변경 승인 없이 매출 계산 로직이나 세율 설정 변경
운영 장애 백업 미흡으로 장애 발생 시 데이터 복구 불가
로그 미관리 누가 언제 무엇을 변경했는지 추적 불가

3. ITGC의 주요 영역

ITGC는 조직마다 세부 항목이 다를 수 있지만, 일반적으로 접근권한 관리, 변경관리, 운영관리, 백업 및 복구, 보안 설정 관리로 나눌 수 있습니다.

영역 설명
접근권한 관리 사용자 계정 생성, 변경, 삭제, 권한 부여가 승인 절차에 따라 이루어지는지 확인
변경관리 프로그램, 시스템 설정, 데이터베이스 변경이 요청, 승인, 테스트, 배포 절차를 거치는지 확인
운영관리 배치 작업, 장애 처리, 로그 점검, 시스템 모니터링이 적절히 수행되는지 확인
백업 및 복구 중요 데이터가 정기적으로 백업되고 실제 복구 가능한지 확인
보안 설정 관리 비밀번호 정책, 관리자 계정, 공유 계정, 원격 접속 설정 등을 점검

4. 접근권한 관리

접근권한 관리는 ITGC에서 가장 기본적이고 중요한 영역입니다. 사용자는 본인의 업무에 필요한 최소 권한만 가져야 하며, 신규 입사, 부서 이동, 퇴사 시 권한이 적시에 반영되어야 합니다.

점검 항목 확인 내용
신규 계정 생성 계정 생성 전에 권한 요청과 승인 기록이 있는가
권한 변경 부서 이동이나 직무 변경 시 권한이 적절히 조정되었는가
퇴사자 권한 회수 퇴사자의 계정이 즉시 비활성화 또는 삭제되었는가
관리자 권한 관리자 권한 보유자가 업무상 필요한 인원으로 제한되어 있는가
정기 권한 검토 사용자 권한 목록을 정기적으로 검토하고 승인하는 절차가 있는가

접근권한 관리의 원칙은 최소권한입니다. 사용자는 업무 수행에 필요한 만큼만 접근할 수 있어야 합니다.

5. 업무분장과 SoD

SoD는 Segregation of Duties의 약자로 업무분장을 의미합니다. 한 사람이 거래 생성, 승인, 지급, 마스터 데이터 변경을 모두 할 수 있다면 오류나 부정이 발생해도 발견하기 어렵습니다.

위험한 권한 조합 위험 설명
거래 입력 + 거래 승인 본인이 만든 거래를 직접 승인할 수 있음
거래처 생성 + 지급 처리 가짜 거래처를 만들고 지급까지 수행할 수 있음
전표 입력 + 전표 삭제 부적절한 전표를 입력한 뒤 흔적을 제거할 수 있음
사용자 생성 + 권한 부여 승인 없이 계정을 만들고 높은 권한을 부여할 수 있음

6. 비밀번호 및 계정 정책

비밀번호 정책은 시스템 접근을 보호하는 기본 장치입니다. 비밀번호가 너무 단순하거나, 공유 계정을 사용하거나, 장기간 변경 없이 사용하면 계정 탈취 위험이 높아집니다.

  • 비밀번호 최소 길이와 복잡도 기준 설정
  • 로그인 실패 시 계정 잠금 정책 적용
  • 공유 계정 사용 제한
  • 장기 미사용 계정 비활성화
  • 관리자 계정 사용 이력 모니터링

7. 변경관리

변경관리는 프로그램 수정, 시스템 설정 변경, 데이터베이스 변경 등이 승인 없이 이루어지지 않도록 관리하는 절차입니다. 특히 회계 시스템의 계산 로직, 승인 흐름, 세율, 계정과목 매핑 변경은 재무 데이터에 직접적인 영향을 줄 수 있습니다.

변경관리는 “누가, 왜, 무엇을, 언제 바꾸었는지”를 추적할 수 있어야 합니다.

변경 단계 확인 내용
변경 요청 변경 사유와 요청자가 명확한가
승인 업무 책임자와 IT 책임자의 승인이 있는가
개발 및 수정 개발 환경과 운영 환경이 분리되어 있는가
테스트 변경 내용이 테스트되고 결과가 문서화되었는가
배포 승인된 변경만 운영 환경에 반영되었는가
사후 확인 배포 후 오류 여부를 확인했는가

8. 개발·테스트·운영 환경 분리

개발자가 운영 시스템을 직접 수정할 수 있으면 무단 변경 위험이 커집니다. 따라서 개발 환경, 테스트 환경, 운영 환경은 분리되어야 하며, 운영 반영은 승인된 담당자만 수행하는 것이 바람직합니다.

환경 역할
개발 환경 프로그램 수정과 신규 기능 개발 수행
테스트 환경 변경 내용 검증 및 사용자 테스트 수행
운영 환경 실제 업무와 회계 데이터가 처리되는 환경

9. 운영관리

운영관리는 시스템이 안정적으로 작동하도록 일상적인 운영 절차를 관리하는 영역입니다. 배치 작업 실패, 인터페이스 오류, 서버 장애, 로그 미점검 등은 업무 중단이나 데이터 누락으로 이어질 수 있습니다.

운영관리 항목 점검 내용
배치 작업 관리 정기 작업이 정상 완료되었는지 모니터링하는가
인터페이스 관리 시스템 간 전송 건수와 금액이 일치하는지 확인하는가
장애 관리 장애 발생 시 기록, 원인 분석, 재발 방지 조치가 이루어지는가
로그 관리 중요 작업 로그가 보관되고 정기적으로 검토되는가
작업 스케줄 관리 중요 작업의 실행 시간, 담당자, 실패 대응 절차가 정해져 있는가

10. 백업 및 복구

백업은 장애나 데이터 손실이 발생했을 때 업무를 복구하기 위한 핵심 통제입니다. 백업을 수행하는 것만으로 충분하지 않으며, 실제로 복구 가능한지 정기적으로 테스트해야 합니다.

  • 중요 데이터의 백업 주기 정의
  • 백업 파일의 보관 위치와 접근권한 관리
  • 백업 성공 여부 모니터링
  • 복구 테스트 수행 및 결과 기록
  • 장애 발생 시 복구 목표 시간과 복구 목표 시점 정의

11. 로그 관리와 모니터링

로그는 시스템에서 발생한 행위를 추적할 수 있는 중요한 증거입니다. 로그가 없으면 누가 언제 로그인했는지, 누가 데이터를 수정했는지, 관리자 권한으로 어떤 작업을 했는지 확인하기 어렵습니다.

로그 유형 확인할 내용
로그인 로그 비정상 시간대 로그인, 실패 로그인, 퇴사자 로그인 여부
권한 변경 로그 누가 누구에게 어떤 권한을 부여했는지
데이터 변경 로그 중요 마스터 데이터나 거래 데이터 수정 이력
관리자 작업 로그 관리자 계정으로 수행한 중요 작업 내역

12. ITGC 감사 증거

ITGC 감사에서는 통제가 실제로 운영되었는지 확인하기 위해 증거를 수집합니다. 단순히 담당자의 설명만 듣는 것이 아니라, 승인 기록, 시스템 설정 화면, 로그, 티켓, 보고서 등을 통해 확인해야 합니다.

영역 주요 증거
접근권한 사용자 목록, 권한 요청서, 승인 기록, 퇴사자 목록, 권한 검토 기록
변경관리 변경 요청 티켓, 승인 내역, 테스트 결과, 배포 기록
운영관리 배치 결과, 장애 티켓, 운영 로그, 인터페이스 대사 자료
백업 복구 백업 성공 로그, 백업 정책, 복구 테스트 결과
보안 설정 비밀번호 정책 화면, 관리자 계정 목록, 로그 설정 화면

13. ITGC 점검 예시

접근권한 점검 예시

1. ERP 사용자 목록 다운로드
2. 인사팀 퇴사자 목록과 비교
3. 퇴사일 이후 활성 계정 확인
4. 관리자 권한 보유자 추출
5. 권한 승인 기록과 실제 권한 비교
6. 예외 사항을 발견사항으로 정리
변경관리 점검 예시

1. 감사 기간 중 운영 배포 목록 확보
2. 각 변경 건의 요청서 확인
3. 승인자와 승인일 확인
4. 테스트 결과 문서 확인
5. 실제 배포 기록과 변경 요청 목록 대사
6. 승인 없는 변경 여부 확인

14. ITGC 발견사항 예시

발견사항 위험 개선방안
퇴사자 계정이 ERP에 활성 상태로 남아 있음 무단 접근 및 데이터 변경 가능성 퇴사 프로세스와 계정 회수 절차를 연계하고 정기 점검 수행
관리자 권한 보유자가 과도함 권한 남용 및 추적 어려움 최소권한 원칙에 따라 관리자 권한 재검토
운영 배포 전 테스트 증거가 없음 오류가 검증 없이 운영 시스템에 반영될 수 있음 테스트 결과 문서화 후 승인된 건만 배포
백업 복구 테스트를 수행하지 않음 장애 시 실제 복구 가능성을 확인할 수 없음 정기 복구 테스트를 수행하고 결과를 기록

15. 역할극으로 이해하는 ITGC

감사인:
퇴사자 계정이 아직 ERP에 활성 상태로 남아 있습니다.

IT 담당자:
실제로 로그인하지 않았으니 괜찮지 않나요?

감사인:
로그인 여부보다 접근 가능성이 남아 있다는 것이 문제입니다.

회계 담당자:
그게 회계 데이터에도 영향을 줄 수 있나요?

감사인:
네. 전표 수정, 거래처 정보 변경, 지급 정보 변경 같은 위험이 생길 수 있습니다.

IT 담당자:
그럼 ITGC에서는 무엇을 확인해야 하나요?

감사인:
계정 생성, 권한 변경, 퇴사자 권한 회수, 관리자 권한, 정기 권한 검토를 확인합니다.

회계 담당자:
결국 ITGC는 시스템 관리가 아니라 회계 신뢰성과도 연결되는군요.

감사인:
맞습니다. 시스템 통제가 약하면 데이터 신뢰성도 약해집니다.

학습 정리

  • 우리 조직의 핵심 시스템은 무엇인가
  • 사용자 계정과 권한은 승인 절차에 따라 관리되는가
  • 퇴사자와 부서 이동자의 권한은 적시에 회수되는가
  • 프로그램 변경은 요청, 승인, 테스트, 배포 절차를 거치는가
  • 배치 작업, 장애, 로그, 백업은 정기적으로 관리되는가
  • ITGC 증거는 감사인이 확인할 수 있도록 남아 있는가

한 줄 핵심 정리
ITGC는 정보시스템이 신뢰할 수 있게 운영되도록 만드는 기본 통제이며, 회계 데이터와 내부통제의 신뢰성을 지탱하는 기반입니다.
