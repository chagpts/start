Data Analytics — SQL과 데이터 분석으로 보는 디지털 감사의 핵심

본 자료는 Digital Audit과 회계감사 실무에서 왜 데이터 분석이 핵심 역량이 되었는지를 설명하는 교육용 콘텐츠입니다. 특히 SQL의 기본 개념, transaction data와 master data의 차이, 이상거래 탐지 방식, 그리고 duplicate payment, 이상 매출 패턴, journal entry anomaly 같은 대표적인 실무 분석 사례를 중심으로 구성했습니다.

핵심 메시지
“샘플 감사는 끝났다. 이제는 전수 분석이다.”

교육 범위

• 왜 감사에서 데이터 분석이 중요해졌는가

• SQL 기본: SELECT, JOIN, GROUP BY

• 데이터 구조 이해: transaction vs master

• 이상거래 탐지 방식

• 실무 예시: duplicate payment, 이상한 매출 패턴, journal entry anomaly

1. 왜 이제는 데이터 분석이 중요한가

예전 감사는 표본을 뽑아 증빙을 확인하는 방식이 중심이었습니다. 하지만 지금의 기업은 ERP와 주변 시스템에서 방대한 양의 데이터를 생성합니다. 매출, 매입, 재고, 전표, 권한, 인터페이스 로그, 배치 실행 결과까지 모두 데이터로 남기 때문에 일부 샘플만 봐서는 전체 리스크를 충분히 보기 어려워졌습니다.

그래서 현대의 감사는 “몇 건을 뽑아 보는 감사”에서 “전체 데이터를 스캔해 위험한 패턴을 찾는 감사”로 이동하고 있습니다. 표본은 여전히 중요하지만, 그 표본을 어디서 뽑아야 하는지를 정하는 데도 데이터 분석이 먼저 필요합니다.

핵심 이해
데이터 분석은 샘플 감사를 대체하는 것이 아니라, 샘플을 더 똑똑하게 고르고, 전수에서 위험 패턴을 찾게 해주는 도구다.

2. Data Analytics가 감사에서 하는 역할

감사에서 데이터 분석은 단순히 예쁜 그래프를 만드는 작업이 아닙니다. 그 목적은 전체 모집단을 이해하고, 이상한 거래를 찾고, 통제가 제대로 작동했는지 검토하고, 사람 눈으로는 놓치기 쉬운 패턴을 발견하는 데 있습니다.

데이터 분석의 역할 감사에서 왜 중요한가
전수 분석 전체 거래를 대상으로 위험 신호를 찾을 수 있음
이상 패턴 탐지 중복지급, 비정상 분개, 이상한 매출 흐름 발견 가능
통제 검증 보조 ITAC나 업무 통제가 실제로 작동했는지 데이터로 확인 가능
표본 선정 고도화 무작위가 아니라 리스크 기반으로 표본 선정 가능
설명 가능성 확보 왜 이 거래가 이상한지 근거를 데이터로 보여줄 수 있음

감사 관점
데이터 분석은 “더 많은 걸 보기 위해” 쓰는 것이 아니라, “진짜 위험한 걸 더 빨리 찾기 위해” 쓰는 것이다.

3. SQL이 왜 중요한가

SQL은 데이터베이스에서 필요한 데이터를 꺼내고, 연결하고, 집계하는 가장 기본적인 언어입니다. Digital Auditor나 데이터 분석 담당자가 Python 같은 도구를 쓰더라도, 근본적으로는 데이터를 어떻게 조회하고 연결할지 이해해야 하는데 그 핵심이 SQL입니다.

회계와 감사 실무에서 SQL이 중요한 이유는 ERP나 데이터 웨어하우스 안의 원천 데이터를 직접 보고, 필요한 기준으로 걸러내고, 이상 패턴을 찾기 위한 가장 기본적인 수단이기 때문입니다.

한 줄 요약
SQL은 데이터를 “볼 수 있게 해주는 언어”이고, 데이터 분석은 그 데이터를 “의미 있게 해석하는 사고방식”이다.

4. SQL 기본 ① SELECT

SELECT는 데이터를 조회하는 가장 기본적인 명령입니다. 감사 실무에서는 어떤 테이블에서 어떤 컬럼을 볼지 정하는 것이 첫 단계입니다. 

SELECT vendor_id, invoice_no, payment_amount
FROM ap_payments;

이 쿼리는 지급 데이터 테이블에서 공급업체, 송장번호, 지급금액을 조회합니다. 아직 분석은 아니지만, 분석을 시작하기 위해 필요한 최소한의 데이터 뽑기라고 볼 수 있습니다.

실무적으로 중요한 것

  • 필요한 컬럼만 가져오기
  • 거래일, 금액, 문서번호, 사용자ID 같은 핵심 필드를 이해하기
  • 어떤 테이블이 원천 데이터인지 구분하기

5. SQL 기본 ② WHERE

WHERE는 조건을 주어 필요한 데이터만 필터링하는 데 사용합니다. 감사에서는 기말 전후 거래, 특정 계정, 특정 사용자, 특정 금액 이상 거래를 볼 때 자주 씁니다. 

SELECT document_no, posting_date, amount
FROM gl_entries
WHERE posting_date BETWEEN '2026-12-25' AND '2027-01-05';

이 쿼리는 기말 전후 전표만 뽑아 cut-off 테스트나 이상 전표 검토에 활용할 수 있습니다.

6. SQL 기본 ③ JOIN

실무 데이터는 한 테이블에 다 들어 있지 않습니다. 거래 테이블과 마스터 테이블, 전표 헤더와 전표 라인, 고객정보와 매출정보가 나뉘어 있는 경우가 많습니다. JOIN은 이런 여러 테이블을 연결해 분석 가능한 형태로 만드는 핵심 기능입니다. 

SELECT p.vendor_id, v.vendor_name, p.invoice_no, p.payment_amount
FROM ap_payments p
JOIN vendor_master v
  ON p.vendor_id = v.vendor_id;

이 쿼리는 지급 거래 데이터와 공급업체 마스터를 연결해 숫자뿐 아니라 공급업체 이름까지 함께 볼 수 있게 합니다.

실무 감각
감사 데이터 분석에서 중요한 질문 중 하나는 “이 숫자가 누구와 관련된 거래인가?”인데, 그 답을 찾게 해주는 것이 JOIN이다.

7. SQL 기본 ④ GROUP BY

GROUP BY는 데이터를 집계하여 패턴을 볼 때 사용합니다. 예를 들어 공급업체별 지급 건수, 사용자별 전표 건수, 날짜별 매출 추이 등을 볼 수 있습니다. 

SELECT vendor_id, COUNT(*) AS payment_count, SUM(payment_amount) AS total_paid
FROM ap_payments
GROUP BY vendor_id;

이 쿼리는 공급업체별 지급 건수와 총지급액을 집계합니다. 특정 공급업체에 지급이 비정상적으로 집중되어 있는지 볼 때 유용합니다.

8. 데이터 구조 이해: Transaction vs Master

데이터 분석을 제대로 하려면 transaction data와 master data를 구분할 수 있어야 합니다. 이 둘을 헷갈리면 분석은 하더라도 해석이 틀릴 수 있습니다.

Transaction Data

실제 거래가 발생한 기록입니다. 주문, 출하, 송장, 지급, 전표, 입고, 재고이동 같은 “움직임”이 여기에 해당합니다.

Master Data

거래가 참고하는 기준 정보입니다. 고객, 공급업체, 자재, 계정, 회사코드, 환율유형, 제품분류 같은 “기준값”이 여기에 해당합니다.

구분 Transaction Data Master Data
의미 실제 발생한 거래 거래가 참조하는 기준 정보
예시 지급전표, 매출전표, 출하건, JE 공급업체 마스터, 고객 마스터, 계정 마스터
감사 질문 무슨 거래가 일어났는가? 그 거래는 어떤 기준값을 참조했는가?

중요한 이유
거래 데이터가 이상해 보일 때, 원인은 종종 거래 자체가 아니라 잘못된 마스터 데이터에 있을 수 있다.

9. 이상거래 탐지란 무엇인가

이상거래 탐지는 단순히 “틀린 거래”를 찾는 것이 아닙니다. 정상 패턴에서 벗어난 거래, 통상적이지 않은 구조를 가진 거래, 부정이나 오류 가능성이 높은 거래를 데이터로 식별하는 작업입니다.

대표적인 이상 신호

  • 같은 공급업체에 같은 금액이 반복 지급됨
  • 월말·야간·휴일에 비정상 전표가 집중됨
  • 특정 사용자만 큰 수동전표를 자주 올림
  • 특정 고객에만 급격한 매출 집중이 발생함
  • 송장번호, 지급금액, 지급일 조합이 비정상적으로 반복됨

핵심 이해
이상거래는 곧 오류라는 뜻은 아니지만, “설명이 필요한 거래”라는 뜻이다. 감사인은 바로 그 설명이 필요한 거래를 찾는 사람이다.

10. 실무 예시 ① Duplicate Payment 탐지

Duplicate payment는 가장 대표적인 감사 데이터 분석 주제 중 하나입니다. 동일한 송장에 대해 두 번 지급되었거나, 조금 다른 형태로 중복 지급된 거래를 찾는 것입니다.

대표 탐지 기준

  • 같은 vendor + 같은 invoice_no + 같은 amount
  • 같은 vendor + 같은 amount + 비슷한 payment_date
  • 같은 bank account + 같은 amount + 같은 reference
SELECT vendor_id, invoice_no, payment_amount, COUNT(*) AS cnt
FROM ap_payments
GROUP BY vendor_id, invoice_no, payment_amount
HAVING COUNT(*) > 1;

이 쿼리는 같은 공급업체, 같은 송장번호, 같은 금액이 2건 이상 존재하는 지급을 찾습니다. 중복지급의 가장 기본적인 탐지 방식입니다.

리스크 신호 가능한 의미
완전히 동일한 지급 두 건 중복 지급 가능성
송장번호는 다르지만 금액·일자가 유사 우회 중복 또는 입력 오류 가능성
같은 계좌에 반복 지급 비정상 지급 패턴 가능성

11. 실무 예시 ② 이상한 매출 패턴 찾기

매출은 단순 금액보다 패턴이 중요할 때가 많습니다. 예를 들어 월말에 매출이 과도하게 몰리거나, 특정 고객에게만 갑자기 매출이 급증하거나, 반품과 상계되는 비정상 매출이 생길 수 있습니다.

대표 분석 포인트

  • 월말 집중 매출
  • 휴일 / 야간 posting
  • 특정 고객 또는 특정 영업사원 집중 패턴
  • 큰 매출 후 바로 credit memo 발생
SELECT posting_date, SUM(revenue_amount) AS daily_revenue
FROM sales_transactions
GROUP BY posting_date
ORDER BY posting_date;

이 쿼리는 날짜별 매출 합계를 보여줍니다. 월말 며칠에 매출이 비정상적으로 치솟는다면 cut-off나 channel stuffing 같은 리스크를 의심할 수 있습니다. 

SELECT customer_id, SUM(revenue_amount) AS total_revenue
FROM sales_transactions
GROUP BY customer_id
ORDER BY total_revenue DESC;

이 쿼리는 고객별 매출 집중도를 봅니다. 특정 소수 고객에만 매출이 지나치게 몰려 있으면 계약조건, 반품조건, 인식시점 등을 추가 검토할 수 있습니다.

실무 감각
매출 분석은 “얼마냐”보다 “언제, 누구에게, 어떤 패턴으로 발생했느냐”를 보는 것이 더 중요할 때가 많다.

12. 실무 예시 ③ Journal Entry Anomaly

Journal entry anomaly는 총계정원장 전표 중에서 통상적이지 않은 패턴을 가진 전표를 찾는 분석입니다. 특히 수동전표, 야간전표, 휴일전표, round number 전표, 특정 사용자 집중 전표는 감사에서 자주 보는 이상 신호입니다.

대표 분석 포인트

  • 수동 JE만 따로 보기
  • 월말 / 분기말 / 연말 집중 여부
  • 휴일 또는 야간 posting
  • 정확히 1억, 5천만 같은 round number
  • 특정 사용자 작성 빈도 과다
SELECT preparer_id, COUNT(*) AS je_count, SUM(amount) AS total_amount
FROM journal_entries
WHERE je_type = 'MANUAL'
GROUP BY preparer_id
ORDER BY total_amount DESC;

이 쿼리는 수동전표를 사용자별로 집계합니다. 특정 사용자가 유독 큰 금액의 수동전표를 자주 올린다면 추가 검토가 필요할 수 있습니다. 

SELECT document_no, posting_date, amount
FROM journal_entries
WHERE je_type = 'MANUAL'
  AND amount % 10000000 = 0;

이 쿼리는 1천만원 단위로 딱 떨어지는 round number 수동전표를 찾는 예시입니다. 물론 모든 round number가 문제는 아니지만, 설명이 필요한 전표를 뽑는 좋은 시작점이 됩니다.

이상 전표 패턴 왜 보는가
휴일 / 야간 전표 통상 처리시간 밖에서 입력된 예외 거래 가능성
수동 대량 전표 자동통제 우회 가능성
Round number 근사값 조정 또는 의도적 조정 가능성
특정 사용자 집중 권한 남용 또는 검토 부재 가능성

13. 데이터 분석을 잘하려면 무엇이 필요한가

좋은 데이터 분석은 SQL 문법만 안다고 되는 것이 아닙니다. 무엇보다 회계 프로세스, 시스템 구조, 데이터 의미를 이해해야 합니다. 즉, “데이터를 꺼내는 기술”보다 “무슨 질문을 할 것인가”가 더 중요합니다.

필요한 역량

  • 회계 프로세스 이해
  • ERP 구조 이해
  • 테이블 간 관계 이해
  • SQL 기본 문법 활용 능력
  • 이상 패턴을 의심하는 감사적 사고
  • 결과를 해석하고 설명하는 능력

중요한 포인트
데이터 분석은 코딩 대회가 아니다. 실무에서는 “어떤 질문을 던져야 위험이 보이는가”가 가장 중요하다.

14. Digital Audit에서 데이터 분석이 강력한 이유

Digital Audit은 시스템이 회계 숫자를 어떻게 만드는지 보는 업무입니다. 그 과정에서 데이터 분석은 단순 보조도구가 아니라 핵심 방법론이 됩니다. 왜냐하면 시스템 통제와 실제 거래 결과를 전체 데이터로 검증할 수 있기 때문입니다.

Digital Audit 질문 데이터 분석이 하는 역할
매출 인식이 제대로 되었는가? 배송일 vs 매출일 전수 비교
중복지급이 있는가? 송장번호, 금액, 날짜 조합 중복 탐지
수동 조정이 과도한가? 수동 JE 패턴 분석
내부거래 제거가 제대로 되었는가? entity 간 거래 매칭 분석
권한 남용 흔적이 있는가? 사용자별 activity 분석

결론
Data Analytics는 Digital Audit의 “눈”이다. 전체 데이터를 보지 못하면 전체 리스크도 보기 어렵다.

Digital Auditor: 표본도 필요하지만, 먼저 전수 데이터에서 중복지급 패턴이 있는지 보는 게 좋습니다.

Financial Auditor: 표본보다 전수 분석이 더 효율적일 수도 있다는 말인가요?

Digital Auditor: 맞습니다. 전체를 먼저 스캔하면 어디가 위험한지 보이고, 그 다음 표본을 더 똑똑하게 고를 수 있습니다.

Financial Auditor: 매출도 그런 식으로 볼 수 있나요?

Digital Auditor: 네. 날짜별 매출, 고객별 집중도, 월말 매출 몰림, 반품 연계 패턴 등을 전수로 볼 수 있습니다.

Financial Auditor: 그럼 수동전표도 전수로 볼 수 있겠네요.

Digital Auditor: 바로 그 부분이 중요합니다. 휴일 전표, 야간 전표, round number 전표를 다 찾아볼 수 있습니다.

Financial Auditor: 이제 왜 데이터 분석이 감사의 핵심 도구가 되었는지 알 것 같습니다.
